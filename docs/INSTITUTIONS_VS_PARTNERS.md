# 机构与伙伴的区别和关系

## 一、核心结论（一句话）

机构（Institution）是“组织主体档案”，伙伴（Partner/Sponsor）是“合作身份与展示分层”；二者有关联，但不是同一个概念。

## 二、区别

### 1. 数据语义不同

- 机构：描述“这个组织是谁”，包括名称、类型、国家地区、简介、官网、Logo 等。
- 伙伴：描述“这个组织以什么合作身份参与本届活动”，例如赞助级别、展示分区、是否首页展示、优先级等。

### 2. 生命周期不同

- 机构：更长期、跨年度可复用的主数据。
- 伙伴：更偏年度/项目期配置，可能随届次、合作方案变化。

### 3. 关系建模不同

- 机构可以连接多个业务对象：活动、嘉宾、知识成果、赛道关系。
- 伙伴信息更偏“合作身份层”，常用于前台合作伙伴分组展示与商务管理。

### 4. 权限与接口侧重点不同

- 机构管理强调结构化关系维护（如关系类型、范围、赛道关联）。
- 伙伴管理强调赞助层级与展示运营（如 tier、排序、是否展示）。

## 三、关系

### 1. 业务关系：同一组织可同时具备两层身份

- 组织先作为“机构”存在（主档案）。
- 在某一届活动中，再被赋予“伙伴/赞助商”身份用于展示与权益管理。

### 2. 前台关系：常常聚合展示

- 前台伙伴页面通常会把赞助商分层与机构展示区块做聚合呈现。
- 用户看到的是统一“合作生态”，但后台语义仍应分层维护。

### 3. 治理关系：主数据与运营数据分离

- 机构承担主数据一致性。
- 伙伴承担运营编排灵活性。
- 这样既能稳定复用，又能快速调整年度策略。

## 四、为什么不建议把两者合并成一个表

- 会把“组织事实”与“年度合作状态”耦合在一起。
- 历史追踪和跨届复用会变困难。
- 权限边界不清：内容编辑与商务运营互相干扰。
- 前台看似简单，后台维护成本和脏数据风险会变高。

## 五、推荐实践

- 保持双模型：机构做主档案，伙伴做合作身份。
- 通过关系字段和聚合查询在前台统一呈现。
- 在后台明确模块边界与权限边界，避免误操作。
